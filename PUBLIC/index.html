Mars Map

HTML5 / CSS3 / leaflet.js / featherlight.js

This web application presents an interactive map of the planet Mars, which I developed using a JavaScript plugin called leaflet.js. While NASA and Google have their own interactive maps of Mars, I wanted to created something that was a bit more stylistically nuanced.

more info

Leaflet.js & Custom Markers

I found leaflet.js to be incredibly useful, especially in terms of how customizable many of the included elements were. The code below demonstrates how via the plugin's JavaScript and CSS files (latter not shown), I was able to deploy my own icons for map markers, plot those markers on the map, and then separate them into marker groups revealable through a layers control element.


              //Creating a custom icon for craters of interest markers

              var craterIcon = L.icon({

                 iconUrl: 'img/craterMarker.png',

                 iconSize:     [32, 32], // size of the icon

                 iconAnchor:   [24, 24], // point of the icon which will correspond to marker's location

                 popupAnchor:  [-16, -30 ] // point from which the popup should open relative to the iconAnchor

              });

              

              //Initializing the craters of interest markers, assigning the custom icon, and binding a pop, from which a relevant lightbox can be accessed

              var bacolorMarker = L.marker([-114, 831], {icon: craterIcon}).bindPopup("<h3>Bacolor Crater</h3><h4>33°N, 241.4°W</h4><h5 data-featherlight='#bacolorLightbox'>more info</h5>")

                 huygensMarker = L.marker([-255, 660], {icon: craterIcon}).bindPopup("<h3>Huygens Crater</h3><h4>13.9°S, 304.4°W</h4><h5 data-featherlight='#huygensLightbox'>more info</h5>")

Featherlight.js & Styled Lightboxes

I wanted to ensure that there was never too much information on screen at a time. If users wanted to enjoy the map without any markers or additional features, they could choose to do so. If however, they wanted to access information about any given marker, they could through a lightbox feature, for which I implemented using featherlight.js. What I loved about this plugin, much like Leaflet, was how easy it was to manipulate the styling. This allowed me to add my own flair to the application, which was one of my primary goals in distinguishing this map from ones currently available.

Sass

This was my first major project for which I developed in a Sass environment. I considered this to be, without a doubt, a game changer in how I approached development. Now I can't live without it. What I find most useful about Sass, apart from being able to designate variables, is how you're able to breakup different aspects of your site styling (layout, typography, etc.) into separate stylesheets. This is incredibly helpful when trying to keep your development environment organized, which can quickly become an issue when it comes to building a responsive site.

Reusable Stylesheets

Since this was my most extensive site yet at the time, I decided to deploy separate stylesheets for templated elements. The navbar and footer were the same across all pages, so I coded their CSS in a separate stylesheet and called them on each HTML page, as opposed to copying that code into the stylesheets for all the different page types. This was before I began using Sass in my development process, so I would approach this differently were I building the site now however, at the time I felt it was an efficient approach to consolidating my code.

more info

Same Function, New Classes, Different Elements

There are essentially three things on this page that change when a user clicks on the asteroid cluster. The ship fires a laser beam, the cluster breaks up, and the text in the instruction line changes. The first two changes are made by applying the same setAttribute function, and passing in arguments on the event handlers. Similarly, the text change is also handled with a reusable function.


              /* function laserBlast():

                  -Accepts ID of HTML elementID

                  -Accepts CSS class name to apply to HTML element

                  -Locates HTML by ID

                  -Changes class of target element

              */

              function laserBlast(elementID, newClass) {

                  document.getElementById(elementID).setAttribute("class", newClass);

              };

              

              /* function newInstruction():

                  -Accepts ID of HTML elementID

                  -Accepts new HTML to be applied to target element

                  -Locates HTML by ID

                  -Changes inner HTML

              */

              function newInstruction(elementID, newInstruction) {

                  document.getElementById(elementID).innerHTML = newInstruction;

              };

riffBites

Photoshop / InDesign / Adobe XD / InVision

riffBites is a music recording/sharing application geared towards musicians looking to work on collaborative projects remotely. It was designed to be more lightweight than what is commonly seen is professional audio software. In leading up to the final prototype, our team conducted stakeholder interviews, and from there, developed user personas and product requirements. We then developed and revised our information architecture before conducting user testing and completing final revisions.

more info

Competitive Analysis

In identifying the white space opportunities for riffBites, our team conducted a thorough competitive analysis. What we learned is that while sharing was certainly not something new to audio recording applications, we saw that many applications and services were convoluted in their user experience. As such, it was at this time that we decided making riffBites as simple and intuitive as possible was paramount. We wanted our users to be able to get in, record, share, and get out with as few screens and clicks as possible.

Full Document

Understanding User Task Flow

Before developing our wireframe, our team elaborated on our user scenarios and goals by producing a user task flow map. This map indicates two distinct usage scenarios with individual needs, but also shows where those needs overlap. This in turn helped us to focus the design of our application.

PDF Format

Project Scope/Requirements

Based on our user scenarios and previously developed user task flow map, we extrapolated a full-fledged project scope document to be referenced and revised as subsequent design and production took place. This allowed us to further identify primary versus secondary functionality, as well as the core ideals to which the application needed to adhere.

Plantastic

Photoshop / Adobe XD / InVision

Plantastic was an event planning application I designed along with a team of 2 others. The process involved stakeholder interviews, competitive analysis, persona development, product requirements, information architecture, a high-fidelity prototype, as well as user testing.

more info

Wireframe & High-Fidelity Prototype

After completing user research, persona development, as well as product requirements, our team first developed a wireframe using Adobe XD, and then a high-fidelity prototype using InVision, a widely used prototyping tool throughout the web/application design industry.

Full Wireframe Full Prototype

Developing Personas and User Goals

In the earlier stages of our user research, we first identified patterns that were prevalent amongst real users who had event planning experience, whether in a professional context, or a personal one. The users were plotted on a variable map, and from there patterns and goals were extrapolated that later informed personas.

Full Document Full Document

Usability Testing

After deploying our high-fidelity prototype, we conducted usability testing. This provided us an excellent opportunity for feedback nearing the end of the design phase for Plantastic. We were able to successfully identify where the product fell short, which lay mostly in missing transitional states, as well as see where certain UI elements were ineffective, such as poor placement of certain buttons. In essence, I would say that more than anything, usability testing for Plantastic made clear the value of agile development, and why working in an iterative process is highly efficient.
